Kernel configuration policy for filter kernels.

Parameterizations of this type encapsulate our kernel-tuning parameters

Kernels can be specialized for problem-type, SM-version, etc. by parameterizing them with different performance-tuned parameterizations of this type. By incorporating this type into the kernel code itself, we guide the compiler in expanding/unrolling the kernel code for specific architectures and problem types.

Template Parameters:
_ProblemDataProblem data type.
_CUDA_ARCHCUDA SM architecture to generate code for.
_INSTRUMENTWhether or not we want instrumentation logic generated
_SATURATION_QUITIf positive, signal that we're done with two-phase iterations if frontier size drops below (SATURATION_QUIT * grid_size).
_DEQUEUE_PROBLEM_SIZEWhether we obtain problem size from device-side queue counters (true), or use the formal parameter (false).
_MIN_CTA_OCCUPANCYLower bound on number of CTAs to have resident per SM (influences per-CTA smem cache sizes and register allocation/spills).
_LOG_THREADSNumber of threads per CTA (log).
_LOG_LOAD_VEC_SIZENumber of incoming frontier vertex-ids to dequeue in a single load (log).
_LOG_LOADS_PER_TILENumber of such loads that constitute a tile of incoming frontier vertex-ids (log)
_LOG_RAKING_THREADSNumber of raking threads to use for prefix sum (log), range [5, LOG_THREADS]
_END_BITMASK_CULL,Iterationafter which to skip bitmask filtering (0 to never perform bitmask filtering, -1 to always perform bitmask filtering)
_LOG_SCHEDULE_GRANULARITYThe scheduling granularity of incoming frontier tiles (for even-share work distribution only) (log)
